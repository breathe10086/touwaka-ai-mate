目前这个项目有助理系统，有驻留技能。但是对话中对OCR的调用，仍然不够丝滑。我们需要反思设计上是不是出了问题。

### 驻留技能的设计目的

一开始设计驻留技能，是为了解决LLM对话中的阻塞现象：有些场景的技能调用无法立刻得到返回，就必须考虑使用另外一个进程处理连接，而把消息用异步的方式桥接回主对话。可以认为是某种子agent。
以下是相关场景

1. 用llm连接一个或多个ssh服务器，你得一边保持连接池，一边继续对话。
2. 主LLM调用另外一个LLM做OCR，做图片创作等
3. 调用第三方API进行数据交互，比如数据库，比如ERP/BI系统的查询。
   这些都是消息发出去无法立刻得到回复的，适合“回调”式访问。
   这方面基本也实现了：将返回结果作为tool调用的记录写入messages并触发LLM继续对话以分析返回结果，但是我们需要有一个地方能够更细致的查询调用过程，如果发生了阻塞，我们需要知道阻塞在哪里。我们不希望另外增加新的表。
   想法：使用messages表，role为特殊的tool，比如tool-R或者其他更好的名字。当然在拼接上下文的时候，这个role还是要转换为‘tool’的，否则上下文搞不清。但是在查询的时候，可以比较明确的知道，messages表里有哪些工具调用，哪些是属于哪个驻留技能的。这里帮我完善。

助理系统实际上就是子agent，不过当前的设计过于臃肿，非常不优雅。委托/返回模式本无问题，但独立于messages另外搞一套，是设计上的严重失误。属于没有捋清楚就开始干了，叠床架屋。

### 当前项目遇到的痛点

我们计划扩展驻留技能，但是当前出发点是为了使得remote-llm更好用。这是有问题的。因为remote-llm是可以被新的子agent系统取代的，因此依照这个需求开展的增强，会影响到正确的设计。

另外一个痛点：很多循环类，由LLM执行，会很难控制。比如多张发票，要求依次调用发票解析技能，并把生成的结果按一定格式解析。
实际上专家系统解决的问题是开放问题，而解决方案解决的问题是封闭问题：固定套路，精确循环，明确的格式和交付标准。因此这样的需求让我们不得不怀念起dify和n8n这种workflow来，虽然灵活性不够，但交付能力足够。

因此我们实际上需要一种能把技能给串起来的技能，如何实现强约束？

在之前的开发中，我们通过约定task的目标、过程、交付标准，以及自我检查来实现这一点。当然这个功能还有很多值得改善的地方，但就目前而言，还需要考察其是否值得信赖。因此，子agent一定不是群聊。主LLM维持着完整上下文负责任务的拆解，而为了完成特定任务的子agent，则带有少量的上下文和有限的工具调用，且每次调用都是全新上下文，它们没有记忆。

### 合同/文档比对

比如我们有两份pdf扫描版合同，每份都有20M大小，这略微有点夸张，但也常见。因此为了提高效率我们需要执行以下操作：

1. 拆分合同到图文混编
2. 对图片依次调用LLM进行OCR
3. 识别图片是插图还是扫描件，如果是插图返回描述，如果是扫描件返回原文
4. OCR把结果嵌入回md文档，并再次整理md检查乱码等。
5. 两份合同都整理好了，开始拆分成section / item结构，然后逐项比对。
6. 生成比对报告

那么我们的架构应该如何演进，以使得可以从解决方案入手，完美实现这个功能？
